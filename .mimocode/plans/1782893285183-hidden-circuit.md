# Plan: Add Southern Cities & Pre-load Historical Building Permits

## Current State
- **Backend is fully on Vercel** (Next.js API routes) — no Railway
- Database: PostgreSQL + PostGIS on Supabase, Drizzle ORM
- Schema has `jurisdictions` table (socrata_domain, resource_id, app_token, column_field_map) and `filings` table
- Vercel cron runs `/api/jobs/poll` daily, fetching from Socrata SODA APIs
- **No jurisdictions exist yet** — the admin page hardcodes `["default"]` as seed IDs
- **No seed script or migration exists** in the project

## Confirmed Working Socrata Datasets

### 1. Austin, TX
- **Domain:** `data.austintexas.gov`
- **Resource ID:** `3syk-w9eu`
- **Records:** ~2.3M total, actively updated
- **Key fields:** `permit_number`, `permit_location` (address), `issue_date`, `latitude`, `longitude`, `permit_type_desc`, `permit_class_mapped`, `status_current`
- **Column field map:**
  ```json
  {
    "address": "permit_location",
    "issued_date": "issue_date",
    "permit_number": "permit_number",
    "latitude": "latitude",
    "longitude": "longitude"
  }
  ```

### 2. Orlando, FL
- **Domain:** `data.cityoforlando.net`
- **Resource ID:** `ryhf-m453`
- **Key fields:** `permit_number`, `permit_address`, `issue_permit_date`, `geocoded_column` (point type with lat/lng), `application_type`, `worktype`
- **Column field map:**
  ```json
  {
    "address": "permit_address",
    "issued_date": "issue_permit_date",
    "permit_number": "permit_number"
  }
  ```
- Note: Orlando uses a `geocoded_column` (point type) rather than separate lat/lng fields — will need special handling in the poll route

### 3. Kansas City, MO
- **Domain:** `data.kcmo.org`
- **Resource IDs:** `cwrz-29jm` (2020), but these are historical annual snapshots — not a live feed
- **Not suitable** for ongoing polling (static annual datasets)

### 4. Collin County, TX (Dallas suburbs)
- **Domain:** `data.texas.gov`
- **Resource ID:** `82ee-gbj5`
- **Records:** ~66K, updated daily
- **Key fields:** `permitnum`, `situsconcatshort` (address), `permitissueddate`, `situscity`, `situszip`
- **Note:** County-level (covers Plano, Frisco, McKinney, Allen, etc.) — covers Dallas suburbs but not Dallas city proper
- **Column field map:**
  ```json
  {
    "address": "situsconcatshort",
    "issued_date": "permitissueddate",
    "permit_number": "permitnum"
  }
  ```

## Implementation Plan

### Step 1: Create seed script `scripts/seed-jurisdictions.ts`

Create a TypeScript script (runnable via `npx tsx`) that:
1. Connects to the database using `lib/db/index.ts`
2. Inserts jurisdiction records for confirmed working cities
3. Pre-loads historical permits by fetching from Socrata APIs with a wide date range

### Step 2: Modify poll route for historical backfill

The current poll route (`app/api/jobs/poll/route.ts`) has two issues:
1. **`$limit=5`** — only fetches 5 records per poll (line 81). For historical seeding, we need to fetch thousands.
2. **New jurisdictions have no watermark** — defaults to 24 hours ago (line 76), which would miss historical data.

**Changes needed:**
- Add a separate `/api/jobs/backfill` route that:
  - Accepts a `jurisdiction_id` and `start_date` parameter
  - Fetches historical records in batches of 1000 (Socrata max per request)
  - Processes each record through the same ingestion pipeline as the poll route
  - Handles geocoding for records without lat/lng
  - Advances the watermark after each batch
  - Stops when no more records are returned
- Update the poll route to increase `$limit` from 5 to 1000 for normal operation

### Step 3: Add backfill API route `app/api/jobs/backfill/route.ts`

```typescript
// POST /api/jobs/backfill
// Body: { jurisdiction_id: string, start_date: string, end_date?: string }
// Fetches historical records in batches, inserts into filings table
// Uses same ingestion logic as poll route (column remapping, geocoding, dedup)
```

### Step 4: Create a one-shot seeding endpoint `app/api/admin/seed/route.ts`

A protected admin endpoint that:
1. Inserts all predefined jurisdictions (Austin, Orlando, Collin County)
2. Triggers backfill for each jurisdiction
3. Returns a status report

This avoids needing a separate script with database credentials — everything runs through the Vercel deployment.

### Step 5: Update admin page

Update `app/admin/page.tsx` to:
1. Replace hardcoded `seedJurisdictionIds = ["default"]` with dynamically fetched IDs from a new `GET /api/jurisdictions` list endpoint
2. Add a "Seed Data" button that triggers the admin seed endpoint

### Step 6: Add jurisdictions list endpoint

Add `GET /api/jurisdictions` route that returns all jurisdictions (needed for the admin page to dynamically list them).

### Step 7: Update `.env.example`

Add `DATABASE_URL` to `.env.example` since the backend now runs entirely on Vercel.
